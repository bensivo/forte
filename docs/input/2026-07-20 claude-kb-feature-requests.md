# Claude Feature Requests

## Overview
A set of feature requests generated by claude itself, with this protocol:

1. Created a new forte vault in a tempdir
2. Added 2 docs, Mirion quarterly earnings reports
3. Told Claude to extract and link entities from the reports - using the /ingest skill
4. Asked claude, "what features of forte would be most useful to you to make this a useful, queryable knowledge base"


## Feature Requests
For general "who is X" / "what's coming next for Y" questions, the priorities shift pretty differently:

1. Entity pages need a synthesized summary, not just a field dump + mention list. forte entity show 7 currently gives me "Erin Schesny — media contact" plus a bare list of doc IDs that mention her. To actually answer "who is Erin Schesny" well, I have to go re-read every linked doc's surrounding text myself each time to reconstruct context. A cached --summary field on the entity (that I or the ingest step writes/updates as new docs mention them) would mean the knowledge compounds instead of resetting per-question.

2. Mention context, not just mention existence. The link just says doc #2 mentions entity #7 — it doesn't say what was said about her. Even a short quote/snippet captured at link time (forte doc link 2 7 --context "media contact, quoted regarding...") would make "who is X" answerable from the entity page alone, without re-fetching full doc

3. Relationships between entities, not just entity-doc links. "Next big things for Mirion" is really a graph question — Mirion → initiatives → people leading them → related companies. Right now entities only link to docs, not to each other, so I can't traverse "Mirion's initiatives" directly; I have to link every doc and manually infer which entities co-occur. An explicit entity-to-entity relation (forte entity link 2 17 --rel "pursuing") would make graph-shaped questions like this a traversal instead of re-reading.

4. Recency-aware retrieval. "What's next" implies weighting newer docs over older ones. Right now nothing on a doc or mention is queryable by date except what's buried in the ingested timestamp — a way to sort/filter an entity's mentions by doc recency would matter a lot for "what's coming" vs "what happened historically."


Of these, #3 (entity-to-entity relationships) is the biggest shift from what forte does today — it's the difference between a tagged document store and an actual knowledge graph, which is what "who is X" / "what's next" questions really want to traverse.
