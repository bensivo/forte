# Interview - Agent Layer Solution Design

- **Topic:** Solution design for embedding the AI agent layer into Forte — making entity extraction, linking, and field extraction runnable natively via the `forte` CLI (`forte agent process`, `forte agent ingest`) instead of being driven by an external agent (the `ingest` Claude Code skill).
- **Interviewee:** Ben (SME / architect, solo developer)
- **Interviewer:** Product management assistant (Claude)
- **Date:** 2026-07-20

## Summary

The task is to collapse the work currently done by the hand-driven `ingest` skill into the `forte` binary itself, so a user with just the CLI + an API key gets entity extraction with zero external-agent harness. Key decisions:

- **Both surfaces coexist.** The deterministic primitives (`schema`, `entity`, `doc`) stay first-class for manual control; the agent is an *additive* automation layer on top, not a replacement.
- **Split by cost/determinism, not by noun.** Anything deterministic and free (doc preprocessing, embedding storage/query, rule-based candidate matching) lives in core `forte`. Anything that burns tokens lives under a dedicated **`forte agent`** namespace.
- **Embedded structured pipeline now (design-doc option A); tool-using agent loop later.** The near-term build is the 5-step structured pipeline. A tool-loop is expected later for the "chat with your knowledge base" feature — and A's internal operations should be a clean, reusable capability layer that the future loop can wrap as tools.
- **Full review flow, but simplified state.** Human review is the quality gate (keeps the graph free of obvious/useless relationships), so `forte agent process` keeps per-change approve/reject. **Stepwise review** (option B): extract → review → link/create → review → field-extract → review. But **all state is in-memory** — no `ingest_changes` table, no persistence, no resume. Ctrl-C loses progress (acceptable). Everything commits at the very end.
- **Commit is best-effort**, not atomic across markdown + SQLite; report successes and failures to the user. Mechanical `doc ingest` happens up front (Ctrl-C leaves the doc ingested; duplicate-ingest provisions make that harmless).
- **Command surface:** `forte doc ingest <path>` (unchanged, deterministic) → `forte agent process <doc-id>` (the LLM workhorse) → `forte agent ingest <path>` (convenience wrapper = ingest + process). One doc at a time. `--yes` (auto-approve) and `--dry-run` (propose without writing) flags.
- **LLM boundary:** first-class structured/JSON output (SDK feature — verify against current Anthropic capabilities when writing the design). Stub at the **low-level `messages()`** boundary so tests exercise real parse/validate and can inject malformed JSON. **Single model** for all steps.
- **Retry policy:** each step retries up to **5 times**, then the whole run fails and nothing commits. Betting on prompt quality for ~98%+ per-step success. Persistence/resume added later only if wasted-cost becomes a real complaint.
- **Embeddings deferred** — rule-based linking only for now; perf hit at scale accepted. Prompts live in source code for now (per-vault prompt override noted as a compelling future feature). **Quote capture is in** (powers the review evidence); field-value provenance is deferred.
- **The `ingest` skill will be thrown away** — a one-off, not even worth using as a reference.
- **Forward-looking constraint:** the app is eventually becoming a **GUI/web app** (upload doc → loaders → proposed entities → Approve button). The agent pipeline must be **decoupled from the CLI/Rich presentation layer** so a web request/response cycle can drive the same core flow.

## Interview Log

Q: The `ingest` skill and the solution-design doc represent two philosophies for the agent layer: (A) Embedded agent — LLM calls live inside the `forte` binary, deterministic and testable, no Claude Code required; (B) External agent — the CLI stays a bag of primitives driven by something like Claude Code, which is what the skill does today. Your framing ("just run `forte doc ingest`") sounds like (A). Is that right, and is (B)/the skill now a stopgap you're replacing — or do you want both to coexist?

A: I do want both - the agent should be available to people who just want a knowledge base with 0 effort, but the cli should be available when they need more manual control (e.g. manually defining entities that they know they care about). Maybe a separate command like `forte agent ...` would be the best way to build all those agent features?

Q: You floated two surfaces: `forte doc ingest` / `forte doc extract` (verbs under the `doc` noun) vs. a dedicated `forte agent ...` namespace. `forte agent ...` frames the LLM work as its own capability and quarantines the "this costs money" commands under one noun; `forte doc extract` frames it as a stage of the document lifecycle but scatters LLM-touching commands across nouns. Which framing do you want — agent-as-namespace or agent-as-doc-stage? And should `forte doc ingest` stay purely mechanical with the agent step always separate, or should the agent path be a flag on ingest so the zero-effort user gets it in one shot?

A: Build as many core primitives that don't cost money per-run into the main pieces of forte, and keep anything that requires an LLM into `forte agent`. For example, doc pre-processing, and embeddings storage and querying can all be non-agent pieces.

Q: When `forte agent ingest <path>` runs the reasoning steps, how does the agent execute internally? (A) Embedded pipeline — hand-written Python making discrete Anthropic SDK calls (extract → structured JSON, pick-link, extract-fields); Forte owns control flow, LLM fills structured blanks; this is the 5-step pipeline in the design doc, deterministic and stubbable. (B) Tool-using agent loop — an agentic loop where Claude is handed the primitive ops as tools and decides what to call; more flexible but non-deterministic and harder to test/bound cost. The design doc chose A. Does hosting it yourself change that call?

A: I think A would be fine for now, for the basic process of ingesting docs and doing entity extraction. However, in the future, we plan on building a nice UI on top of this with a very free-text "chat with your knowledge base", and that's almost certainly going to require a tool-using agent loop.

Q: solution-design.md specs a heavy review flow: every proposed change persisted to `ingest_changes` and walked through an approve/reject TUI, `--yes` to bypass, resumable re-runs. But your zero-effort framing sounds like auto-approve by default. Which is the build? (A) Full review flow with persistence + TUI + resume; (B) Auto-apply, review after (just do it, print a summary, correct afterward), maybe with `--dry-run`; (C) Both, phased (B now, TUI follow-up). And if B/C, do you still want the `ingest_changes` table now so the reviewable flow can be layered on without a migration?

A: Do the full review flow now. I know it's more scope, but reviewing output of extraction is going to be critical in making the knowledge base well-groomed and useful. Instead of extracting a ton of obvious, non-useful relationships.

Q: With stepwise dependencies (link operates on extracted entities; field extraction on linked-or-created entities), review can interleave: (A) Generate-all-then-review — run all 5 steps, persist everything, then one TUI session; wastes tokens on candidates about to be rejected. (B) Review-between-steps — extract → review → link/create only approved → review → field-extract survivors → review; no wasted calls but stop-start and resume must track the stage. (C) Hybrid — generate all but make the TUI dependency-aware. Which sequencing — A, B, or C?

A: B.

Q: With stepwise review, an ingest run is a state machine (extracting → reviewing-entities → linking → ... → done). If the user Ctrl-C's or a call fails, resume has to pick up where it left off. I need nailed down: (1) where does the stage live — `documents` row, `ingest_changes`, or a separate `ingest_runs` table? (2) what makes a proposed change idempotent across re-runs — dedupe by (doc_id, kind, payload), or never re-run a step that already has persisted proposals? (3) rejected changes — does re-running re-invoke the LLM and filter against past rejections, or does a completed step never re-run? My instinct: resume is stage-driven, not re-generative. Does that match?

A: Let's just keep all state in memory until the final step, where everything is 'committed' to the kb. If someone Ctrl-C's mid-way, they lose all their progress, which is fine by me.

Q: Two things about the final commit: (1) Atomicity — when the run writes all approved changes to markdown + SQLite, all-or-nothing (one txn + all files, roll back on any failure) or best-effort sequential? (2) The doc-copy step — is raw-copy/text-extraction still a separate up-front deterministic step (`forte doc ingest` registers the doc, then the agent does in-memory review-and-commit), or does the agent own the whole thing end-to-end including the copy, so even the raw copy isn't persisted until review finishes? The distinction: up-front registration means Ctrl-C leaves a doc with no entities (harmless, re-runnable); deferring everything means Ctrl-C leaves zero trace.

A: Atomicity, just do best effort. And output [successes] and failures to the user. Mechanical doc ingestion can happen up front. So a Ctrl-C leaves it ingested. It's no big deal because we already have provisions in place for duplicate ingestion.

Q: Given the split, here's the command set I think falls out — tell me what's right/wrong/missing: `forte doc ingest <path>` (unchanged, deterministic — copy, extract text, register, no LLM); `forte agent extract <doc-id> [--yes]` (new core flow on an already-ingested doc: extract → review → link → review → field-extract → review → commit); `forte agent ingest <path> [--yes]` (convenience wrapper = doc ingest then agent extract). Decisions inside: (1) one doc only, or batch (`--all-unextracted`/folder) for the "20 meeting notes" journey? (2) is `--yes` the only non-interactive path, or also a `--dry-run` that proposes without writing? (3) is `extract` the right umbrella verb even though only step 2 is literally extraction, or something like `process`/`analyze`?

A: One doc only for now. `--dry-run` would be useful, thanks for that suggestion. `forte agent process` is good.

Q: The part that determines whether this is testable and cheap — the LLM boundary. (1) Structured output mechanism: (a) tool-use/function-calling with an input schema per step, (b) prompt-for-JSON and parse with bounded-retry-on-malformed, (c) a first-class structured-output/response-format helper if the SDK exposes one? (2) The stub boundary: a high-level `LLMClient` with one method per step (returns already-parsed structures), or a lower-level `complete(messages) -> text` boundary the stub fakes at the raw-completion level? (3) Model config: single model for all steps, or per-step overrides now?

A: 1. I think C is best. Many LLM SDKs / APIs now support JSON output as a first-class feature, and so we can pretty reasonably rely on that. We expect to be able to use that feature. 2. The stub boundary should be more like `messages()`, so that we can test what happens if the JSON output is invalid (like missing fields or typos, or wrong enums). 3. For now, let's do a single model for all steps.

Q: Scope boundary for this task. You said embeddings storage/query and vector matching are core infrastructure, but the design doc defers the entire embeddings decision (local sentence-transformers vs. hosted Voyage) and defers `forte entity search` with it. (1) Are embeddings in scope for this task at all, or does `forte agent process` ship with only rule-based candidate matching (exact/alias/normalized), with vector discovery + `forte entity search` a future effort? (2) Does rule-only candidate discovery meaningfully hurt linking quality, or is exact+alias+normalized enough with the LLM covering fuzzy cases within that set? My assumption: embeddings stay deferred; this task is the LLM pipeline + rule-based linking only.

A: Let's say embeddings are deferred right now. And we'll accept the hit on performance, especially at larger scales. For now, I'm more interested in getting the core of the LLM Agent pipeline working, and we'll likely need tons of tweaks and enhancements - like adding an embeddings database, rewriting prompts, and playing with different models - to get an acceptable level of performance.

Q: Two related things. (1) Prompt management — you expect to rewrite prompts a lot. Where should prompts live: (a) inline in Python, (b) external prompt files shipped with the package, editable without code changes, (c) overridable per-vault (ship defaults, let `.forte/` drop in custom prompts)? (2) Quote/provenance capture — the design doc says each mention persists the supporting quote the LLM cited (powers the review TUI, explains why an entity was linked) and field values record their source doc. Does `forte agent process` capture these from the start (extract returns `{name, schema, supporting_quote}`, fields tagged with source doc), or is that a nice-to-have to defer? My lean: (b) external files now, and yes capture quotes/provenance since the review rationale requires showing evidence.

A: In source code for now. Though overridable per vault is a neat feature idea, because it will help people cater entity extraction to the kinds of docs they expect to put in that vault. Let's do quote capture. It doesn't overly complicate the implementation right now, and the value-add to the user is great, especially for larger docs the user doesn't know a ton about.

Q: A couple of loose ends. (1) Fate of the `ingest` skill once `forte agent process` does natively what it does by driving primitives: (a) delete it, (b) keep as a fallback for environments without an API key but with Claude Code, (c) repurpose it to call `forte agent process`? (2) Cost/usage visibility — should `forte agent process` report token usage / estimated cost per run, supporting the "keep per-doc cost reasonable" NFR? (3) Field-value provenance — since we're capturing quotes anyway, should field values also carry a source-doc reference now, or defer that (just the mention-level quote this iteration)?

A: We'll throw away the ingest skill. It was a one-off that hasn't been finely tuned. I would even hesitate to use it as a reference. Yeah, let's do cost reporting. That's a cool feature. For now, it's okay not to store field value sources.

Q: Final confirming question. (1) Retry/failure policy in the in-memory flow — does "bounded retries, then fail" still hold: a step's LLM call retries N times (transport error or unparseable/invalid JSON), and if it still fails the whole run aborts with a clear error and nothing is committed (since commit is at the end)? Just confirming malformed-JSON retries rather than hard-failing on first bad parse. (2) Anything I missed — the review TUI's exact presentation (assuming it inherits the doc's Rich approve/reject, one at a time, no inline edit); handling a step that returns zero results (skip to done); whether `--yes` + `--dry-run` combine. Happy to leave those to design-doc defaults unless one matters.

A: Just retry each individual step up to 5 times, then fail. I expect we can engineer the prompts well enough to get a good (98%+) level of success with 5 retries on each step. If it becomes a problem in the future, lots of people complain about wasted money due to failed steps, we'll either just work on that step itself, or add persistence and resume later. Just know we're eventually going to turn this entire app into a GUI application, and the whole agent flow will have a different presentation layer (a web app, where users upload the doc, sees loaders, and then the proposed entities, and clicks an 'approve' button). So keep that in mind.
